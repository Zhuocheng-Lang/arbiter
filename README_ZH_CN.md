# arbiter

> 一个面向 Linux 的非侵入式进程优先级管理器，作为现代 `sched_ext`（scx）调度器的辅助工具。

Arbiter 与 scx 调度器协同运行，根据社区维护的规则集，对每个进程的 `nice`、`cgroup` 和 `oom_score_adj` 进行调整；当规则指定了 cgroup 目标时，还会将 `ionice` 语义映射到 cgroup v2 `io.weight`。它几乎完全兼容 [Ananicy-cpp](https://gitlab.com/ananicy-cpp/ananicy-cpp) 的规则格式，并为 scx 环境提供了额外扩展。

## 为什么选择 arbiter？

### 一句话概括

现代 scx 调度器（如 `scx_lavd`、`scx_bpfland` 等）已经接管 CPU 调度。传统 CFS 优先级调节工具（如 `Ananicy` 系列）因此大多失去了作用。Arbiter 则保留 `nice`、`ionice` 和 `cgroup` 这些外围信号，通过调整权重为 scx 调度器提供辅助。

### 为什么传统调度器越来越不够用

传统调度器真正的问题，并不在于它们曾经无用，而在于它们诞生于一个默认前提仍然成立的时代：内核负责决定一切，用户空间只能在既定秩序里微调参数。CFS 代表的是这种范式下最成熟的一代产物。它试图用统一的公平模型容纳所有进程，让桌面交互、后台服务、编译任务与游戏负载在同一套逻辑里竞争 CPU。这在过去是工程上的胜利；但在今天，它越来越像一种高明却僵硬的折中。

现代 Linux 负载已经分裂成彼此矛盾的诉求：有人要最低延迟，有人要最高吞吐，有人要前台丝滑而后台安静，还有人要求系统能根据硬件拓扑与任务形态动态偏置。固定的内核调度策略不再足以覆盖这些差异，于是 `sched_ext` 出现了。它不是在旧秩序上继续打补丁，而是把“调度策略本身”变成可替换、可实验、可演进的对象。当 `scx_lavd`、`scx_bpfland`、`scx_layered` 这样的调度器开始接管 CPU 决策时，传统那套围绕 CFS 设计的优先级工具就注定会退场；时代并没有否定它们的价值，只是收回了它们曾经赖以生效的舞台。

### 为什么基于 `Ananicy` 的社区规则依然有意义

但调度器的更替，并不意味着经验会被一并清空。社区多年积累下来的 `Ananicy` 规则，真正珍贵的部分从来不只是“替 CFS 调几个 nice 值”，而是它记录了一个更难被重建的知识库：哪些进程属于游戏，哪些是浏览器，哪些是编译器，哪些工作负载应该更激进地争取响应，哪些应该主动后退，把资源让给前台。这些判断不是某一种调度器的专属知识，而是长期实践沉淀出的系统经验。

进入 scx 时代后，这些规则并没有失效，它们只是改变了意义。它们不再直接替调度器做最终裁决，而是为调度器提供可信的信号来源：`nice` 会继续影响权重，`ionice` 与 `oom_score_adj` 仍然定义系统在压力下的取舍，`cgroup` 也依旧是组织工作负载层级的重要边界。换句话说，旧规则保留下来的不是过时的技巧，而是对进程身份与资源意图的判断能力。真正需要更新的，不是这批社区知识本身，而是把它们重新接入现代调度生态的方式。

### arbiter 在其中的作用

Arbiter 的意义，正是在这道断层之间搭起一座可落地的桥。它不试图从 scx 手中夺回调度权，也不把自己伪装成另一个调度器；它所做的，是把社区规则里那些关于“这个进程是谁、它应该争取多少响应、又该在多大程度上让位于前台”的判断，重新翻译成 scx 仍然会读取、仍然会尊重的提示信号。`nice` 不是为了怀念 CFS，而是为了让 scx 调度器据此推导权重；`ionice` 与 `oom_score_adj` 不是历史遗留物，而是系统在 IO 压力与内存紧张时仍然有效的取舍语言；`cgroup` 则继续承担着组织工作负载边界、向分层调度器表达结构意图的职责。

因此，Arbiter 既不是对旧时代的缝补，也不是对新调度器的替代。它更像一个纪律严明的解释器：上游继承 Ananicy 社区多年积累的进程知识，下游对接 scx 暴露给用户空间的参数层。调度策略已经更换，但关于工作负载身份的判断仍然值得被保留；Arbiter 的工作，就是让这些判断在 scx 时代继续生效，而且以一种更诚实、更现代的方式生效。

## 功能

- **继承现成的社区规则资产** — 现有的 `.rules` 与 `.types` 文件可以直接复用；`.cgroups` 文件会被提示并忽略
- **事件驱动，而非周期轮询** — 通过 netlink 连接器捕获新进程事件，以近乎即时的速度完成匹配与应用
- **感知 scx 的策略切换** — 检测当前活动的 scx 调度器，并据此选择更合适的应用路径
- **面向 cgroup v2 的优先级信号与工作负载放置** — 直接支持 `nice` 和 `oom_score_adj`；当规则选择了 cgroup 目标时，会把 `ionice` 映射到 `io.weight`。同时可将进程放入指定层级，并为 `scx_layered` 提供适配路径
- **保持兼容的扩展能力** — 在兼容既有规则格式的前提下，允许使用 arbiter 自身的附加字段做更细粒度控制
- **低侵入、低空转开销** — 使用 Rust 编写，以事件驱动为核心，在空闲时几乎不消耗额外 CPU

## 规则语义

- Arbiter 会先加载每个配置目录中的 `.types`，再加载 `.rules`，因此规则条目可以继承类型预设。
- 规则匹配采用“首次命中即生效”的策略：第一个匹配当前进程上下文的已解析规则会被应用，后续规则不再继续检查。
- `.cgroups` 文件会被明确警告并忽略，因为 Arbiter 当前只建模 cgroup 放置和可选的 `cgroup_weight`，并不实现 systemd 风格的完整 cgroup 策略层。
- `cgroup_weight` 是 cgroup v2 内部的相对份额，只有在规则同时选择了 cgroup 路径时才有意义。
- 未识别字段会被警告并忽略，这样解析器可以对带有额外元数据的规则文件保持前向兼容。

## 要求

- Linux 内核 ≥ 6.12（支持 `sched_ext`；Arbiter 在没有 scx 的情况下也可工作）
- 拥有 `CAP_NET_ADMIN` 和 `CAP_SYS_NICE` 权限，用于进程事件访问和 `nice` 调整（或以 root 身份运行；cgroup 写入仍需要相应的 cgroup 访问权限）
- cgroup v2 统一层级

## 与 Ananicy-cpp 的关系

在 scx 环境里，Arbiter 的定位是 Ananicy-cpp 的**替代品**，而不是并行共存的伙伴。两者同时运行时，会试图对同一批进程重复写入 `nice` 等属性，结果不是协作，而是相互覆盖；如果决定使用 Arbiter，就应当停用 ananicy-cpp。

但被替换的只是旧的执行方式，不是社区多年积累下来的规则知识。来自 [ananicy-rules](https://github.com/CachyOS/ananicy-rules) 等仓库的 Ananicy-cpp 规则文件，本身就是 Arbiter 要继承的上游资产：它们继续负责描述“谁是谁、谁该更靠前、谁该主动退后”，而 Arbiter 负责把这些判断接入 scx 时代仍然有效的参数层。这不是兼容性上的附赠，而是项目一开始就明确追求的设计目标。
